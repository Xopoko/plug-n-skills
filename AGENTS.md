# Agent Guidance

This repository is the canonical source tree for Xopoko's Plug'n Skills agent
plugins. Keep it public-safe, portable across machines, and usable from Codex,
Claude Code, and Cursor. The tree must never be locked to a single agent.

AGENTS.md is the working contract for automated contributors. Prefer changing
this file when repository rules change, not when a one-off task needs local
context.

## Operating Principles

- Treat the Git checkout as the source of truth. Runtime caches, local
  marketplaces, and installed copies are derived outputs.
- Keep repository-facing text in English and ASCII unless a file already uses a
  different character set for a clear reason.
- Keep examples generic: no private project names, credentials, local absolute
  paths, private names, tenant names, or machine-specific assumptions.
- Make source changes before install changes. Validate the source tree before
  refreshing any local agent state.
- Prefer focused skills plus references over broad catch-all skill text.
- Keep deterministic behavior in scripts when a workflow needs repeatable
  checks, parsing, scoring, packaging, or installation.
- Reference bundled plugin scripts from skills through the `$PLUGIN_ROOT`
  convention (the host's plugin-root variable when defined, otherwise the
  skill folder's `../..`); never depend on the agent's current working
  directory.
- Do not overstate capability. Describe what the plugin actually provides and
  what validation proves.

## Repository Shape

- `plugins/<plugin-name>/` contains editable plugin source.
- `plugins/<plugin-name>/.codex-plugin/plugin.json` is the Codex manifest.
- `plugins/<plugin-name>/.claude-plugin/plugin.json` is the Claude Code
  manifest.
- `plugins/<plugin-name>/skills/` contains one or more `SKILL.md` entrypoints.
- `plugins/<plugin-name>/references/` contains longer source ledgers,
  scorecards, contracts, runbooks, or operating notes that should not live in
  hot skill files.
- `plugins/<plugin-name>/scripts/` contains deterministic helpers, validators,
  packagers, probes, or reporting tools.
- `plugins/<plugin-name>/assets/` contains icons and other plugin-owned media.
- `.claude-plugin/marketplace.json` is the published Claude Code marketplace for
  this repository.
- `.agents/plugins/marketplace.json` is generated local Codex install state.
  Keep it ignored and out of commits.

## Source And Runtime Boundaries

- Create and modify plugins only under `plugins/<plugin-name>/` when working in
  this repository.
- Do not create source trees under `$HOME/plugins`, `$HOME/.codex`,
  `$HOME/.claude`, or cache directories for repository work.
- Treat `$HOME/.codex/plugins/cache/...` as runtime cache only. Never edit cache
  copies directly.
- Treat `$HOME/plugins/<plugin-name>` as a legacy compatibility surface unless a
  task explicitly asks for that destination.
- Keep local scratch output in ignored locations such as `research/`,
  `skill-synthesis/`, `docs/superpowers/`, or `tmp/` until it is intentionally
  distilled into source.
- Do not commit generated bytecode, dependency folders, build output, local
  marketplace state, or temporary research corpora.

## Plugin Authoring Rules

Each plugin should be understandable as an installable unit from its directory
alone. When adding or materially changing a plugin:

- Keep the directory name, Codex manifest `name`, and Claude manifest `name`
  aligned.
- Keep plugin descriptions concrete and searchable. Include the domain, the
  main workflows, and any important platform or tool names.
- Put routing and action instructions in `skills/`.
- Put long evidence, source notes, compatibility matrices, scorecards, and
  deeper contracts in `references/`.
- Put executable checks and repeatable transforms in `scripts/`.
- Put icons and other media in `assets/` and reference them from manifests only
  when the target agent supports that field.
- Keep plugin-local tests near the plugin when behavior is complex enough to
  regress.
- For session and log parsers, use public-safe synthetic fixtures for every
  supported record-shape variant; never copy raw host rollout excerpts into
  tracked tests.
- Prefer additive, backwards-compatible changes unless a rename or removal is
  explicitly part of the task.

## Skill Authoring Rules

`SKILL.md` files are hot-path context. They should route the agent quickly and
then point to heavier material only when needed.

- Start with clear trigger guidance: when to use the skill and when not to use
  it.
- Keep operational steps concise, ordered, and testable.
- Reference plugin-local scripts instead of reprinting long command sequences.
- Reference `references/` files for long policies, source inventories, tables,
  and examples.
- Avoid local secrets, personal paths, private repo names, or environment values
  in skill text.
- Keep failure modes explicit when a workflow depends on external tools,
  credentials, network access, or generated state.
- Update token tables in `README.md` with `scripts/token-report.py` when skill
  count or skill content changes materially.

## Manifest And Marketplace Rules

Manifests are publication artifacts, not scratch files.

- Keep `.codex-plugin/plugin.json` valid and aligned with the plugin directory.
- Keep `.claude-plugin/plugin.json` valid and aligned with the Codex manifest.
- Keep shared manifest fields (name, description, base version, author,
  license, keywords) identical across both manifests; the plugin validator
  enforces this parity.
- Cursor needs no manifest: it consumes `skills/` directly via
  `scripts/install-cursor-skills.py`.
- Keep repository URLs pointing at the public source location for this checkout.
- Keep `.claude-plugin/marketplace.json` in sync with the actual plugin set.
- Keep `.agents/plugins/marketplace.json` ignored; regenerate it through the
  installer when local Codex state is needed.
- When renaming, adding, removing, or repositioning a plugin, update every
  surface listed in the change matrix below.

## Change Matrix

When a plugin is added, removed, renamed, or materially repositioned, update:

- root `README.md`;
- `plugins/README.md`;
- `.claude-plugin/marketplace.json`;
- `plugins/<plugin-name>/.codex-plugin/plugin.json`;
- `plugins/<plugin-name>/.claude-plugin/plugin.json`;
- `scripts/validate-repository.py`;
- `scripts/tests/test_repo_structure.py`;
- installation docs if the install model changes;
- token tables if skill counts, skill descriptions, or skill bodies changed.

When the repository brand, slug, or marketplace name changes, also update:

- clone and install commands in `README.md`;
- Claude marketplace commands in `docs/ARCHITECTURE.md`;
- repository URLs in plugin manifests;
- generated visual assets and their rendering scripts;
- repository hygiene checks that guard against stale names.

When only documentation changes, still run the repository validator before
claiming the repo is publication-ready.

## Local Installation Flow

When the task asks to create, synthesize, or improve a plugin while this
repository is the active workspace:

1. Write the source change under `plugins/<plugin-name>/`.
2. Validate from this repository.
3. If the plugin should be usable on this machine, refresh the active agent's
   install state from the repository source: Codex via
   `scripts/install-codex-plugins.py --plugin <plugin-name>`, Cursor via
   `scripts/install-cursor-skills.py --plugin <plugin-name>`, Claude Code via
   its marketplace tooling against this checkout.
4. Verify visibility with the matching helper and `--check-only`.

For source-only work, stop after validation and state that no global
install/cache refresh was performed.

## External User Behavior

These repo-local rules apply only when this repository is the active workspace
or an explicit destination. If a user runs an installed plugin without this
repository, new requested plugins or skills should be delivered to that user's
normal global agent surface unless they explicitly select another source
repository.

## Validation

Run these before committing publication-facing changes:

```bash
python3 scripts/validate-repository.py
python3 scripts/install-codex-plugins.py --dry-run
python3 scripts/install-cursor-skills.py --dry-run --dest tmp/cursor-skills-preview
python3 -m unittest discover -s scripts/tests
```

If the local host intentionally disables platform-specific plugins, pass
repeated `--exclude-plugin <plugin-name>` to install or check commands instead
of reinstalling those plugins.

For install-affecting changes, also run a targeted install and visibility check:

```bash
python3 scripts/install-codex-plugins.py --plugin <plugin-name>
python3 scripts/install-codex-plugins.py --plugin <plugin-name> --check-only
```

For script or validator changes, run the relevant tests:

```bash
python3 -m unittest discover -s scripts/tests
python3 -m unittest discover -s plugins/codex-cli/tests
```

For README token-table changes, regenerate or verify the report:

```bash
python3 scripts/token-report.py --format json
```

Validation failures caused by real source files are blockers. Ignored local
scratch directories should stay outside the validation surface.

## Git Hygiene

- Inspect `git status --short --ignored` before editing and before finishing.
- Do not revert unrelated user changes.
- Do not stage ignored runtime state, caches, generated dependency folders, or
  scratch directories.
- Keep commits focused on the requested repository change.
- Before commit, run `git diff --check` and search the tracked tree for stale
  repository names, private terms, and unsupported grand claims.

## Completion Report

When finishing a change, report:

- what source files changed;
- which validation commands ran and whether they passed;
- whether local Codex, Claude, or Cursor install state was refreshed;
- whether changes were committed or left in the working tree.
