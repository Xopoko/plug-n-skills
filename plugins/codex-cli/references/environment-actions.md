# Codex Environment Action Reference

Use this reference with `skills/codex-environments/SKILL.md` when an action
needs long-running process handling, port cleanup, or deeper editing rules.

## Long-Running Actions

For dev servers, the action should either stay alive and stream logs, so the
service remains running while the action is active, or explicitly detach a
background process and tell the user how to stop it.

Prefer staying alive for Codex Run actions. A detached process is easy to leave
running unintentionally.

Avoid background jobs that write directly to the interactive TTY. Shells can
suspend them with `suspended (tty output)`. Redirect server output to a log file,
then `tail -f` that file in the foreground if live logs are useful.

Use readiness checks before opening browsers or simulators:

```bash
until curl -fsS --max-time 2 "http://127.0.0.1:${PORT}/status" >/dev/null 2>&1; do
  if ! kill -0 "$SERVER_PID" 2>/dev/null; then
    tail -n 80 "$LOG_FILE"
    exit 1
  fi
  sleep 1
done
```

When cleaning ports, first try normal termination, then escalate only for known
local dev-server ports:

```bash
PIDS="$(lsof -tiTCP:"$PORT" -sTCP:LISTEN 2>/dev/null || true)"
if [ -n "$PIDS" ]; then kill $PIDS 2>/dev/null || true; fi
sleep 1
PIDS="$(lsof -tiTCP:"$PORT" -sTCP:LISTEN 2>/dev/null || true)"
if [ -n "$PIDS" ]; then kill -9 $PIDS 2>/dev/null || true; fi
```

Do not kill broad process names unless they are scoped to the current project or
port.

## Editing Rules

Before editing:

- inspect the existing environment file;
- inspect any script the action currently calls;
- identify whether the file says it is autogenerated.

If a file says `THIS IS AUTOGENERATED`, still follow the user's direct request,
but keep the patch minimal and be ready for future regeneration. Prefer changing
the script called by the action when possible.

Use `apply_patch` for manual edits. Preserve unrelated user changes.

When adding scripts:

- put them in the project, usually `scripts/`;
- make them executable if intended to be run directly;
- keep output concise and operational;
- document log paths in the script output.

When deleting an action:

- remove only the requested `[[actions]]` block;
- remove helper scripts only when no remaining action or documented workflow uses them;
- do not remove `.codex/` wholesale unless the user explicitly asks.

When renaming an action, change only `name` unless command behavior or icon also
needs to change.

## Verification

After edits, validate at the right depth:

- TOML syntax: parse with an available TOML parser if one exists, or inspect carefully when no parser is available.
- Script syntax: run `bash -n scripts/name.sh`.
- Executability: run `chmod +x scripts/name.sh` for direct scripts.
- Smoke: run the action command or script long enough to prove startup, readiness, and expected UI/browser/simulator behavior.
- Cleanup: stop any process started during verification unless the user asked to leave it running.

For long-running scripts, use a bounded smoke run, verify readiness in another
check, then terminate the process you started.

## Common Failure Modes

- `suspended (tty output)`: background process wrote to TTY. Redirect logs and foreground `tail -f`.
- `event not found` in zsh: pasted command contains `!`. Put fragile shell logic in a script.
- Dev client cannot load from `127.0.0.1`: Metro/dev server exited or was killed after opening the app. Keep the server process alive.
- Port says it is running another app: kill only the listener on the intended port, then wait until the port is clear.
- Action works in shell but not Codex app: replace relative paths with absolute paths and avoid interactive prompts.
